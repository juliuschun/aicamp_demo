## 안건 1: 현재 업무 프로세스 문제점

**김민수**: 회사가 20명이 되면서 예전처럼 ad-hoc으로 일하기 어려워졌어요. 프로세스를 좀 정비해야 할 것 같은데, 각 팀에서 느끼는 문제점이 뭐가 있을까요?

**이서연**: 기획팀 입장에서는 요청 사항이 여러 채널로 들어와서 혼란스러워요. Slack으로 오기도 하고, 이메일로 오기도 하고, 구두로 요청하시기도 하고요.

**정우성**: 개발팀도 마찬가지예요. 갑자기 "이거 급해요"라고 들어오는 요청이 많아서 스프린트 계획이 흐트러지는 경우가 많아요.

**송유진**: 마케팅팀은 다른 팀에 요청한 게 어디까지 진행됐는지 추적하기 어려워요. 물어봐야 알 수 있어서요.

**윤재석**: 영업팀은 고객 요청사항을 제품팀에 전달하는데, 반영 여부를 피드백 받기가 어려워요. 고객한테 뭐라고 답해야 할지 모르겠어요.

### 결정 사항
- 주요 문제: 요청 채널 분산, 우선순위 불명확, 상태 추적 어려움
- 해결 방향: 중앙화된 요청 시스템 + 명확한 워크플로우 정의

### 액션 아이템
- [ ] 이서연: 현재 요청 유형 및 빈도 분석 (9/10까지)
- [ ] 정우성: 개발 요청 처리 현황 정리 (9/10까지)

## 안건 2: 요청 관리 시스템 도입

**김민수**: 우리가 TaskFlow 만드는 회사인데, 정작 내부에서 요청 관리를 잘 못하고 있네요. 자체 제품을 더 활용해볼까요?

**이서연**: 사실 TaskFlow 내부 워크스페이스가 있긴 한데, 프로젝트 단위로만 쓰고 있어요. 요청 관리용 보드를 새로 만들면 좋을 것 같아요.

**송유진**: 좋아요. "요청 티켓" 같은 개념으로 만들면 되겠네요. 누가 요청했고, 담당자가 누구고, 상태가 어떤지 다 보이게요.

**정우성**: 개발 요청은 JIRA를 쓰고 있는데, 이중 관리가 될 것 같아요.

**이서연**: 기획 확정 전 요청은 TaskFlow로 관리하고, 개발이 확정되면 JIRA로 넘기는 건 어떨까요? TaskFlow → JIRA 연동도 가능하니까요.

**김민수**: 그렇게 하죠. 그리고 우리 제품 먹는 것도 중요하니까(dogfooding) 이 기회에 더 적극적으로 써봅시다.

### 결정 사항
- TaskFlow 내부 워크스페이스에 "요청 관리" 보드 신설
- 요청 → 검토 → 승인 → 진행 → 완료 5단계 워크플로우
- 개발 승인된 건은 JIRA 연동으로 자동 생성

### 액션 아이템
- [ ] 이서연: 요청 관리 보드 설계 및 생성 (9/12까지)
- [ ] 정우성: TaskFlow → JIRA 연동 설정 (9/15까지)
- [ ] 이서연: 팀별 요청 템플릿 작성 (9/12까지)

## 안건 3: 회의 문화 개선

**오수현**: 최근 설문에서 "회의가 너무 많다"는 의견이 많았어요. 하루에 회의만 3-4개씩 있는 날도 있더라고요.

**김민수**: 저도 느껴요. 회의가 많으니까 실제 일할 시간이 부족해지고요.

**송유진**: 그리고 회의 목적이 불명확한 경우도 있어요. 그냥 "논의해요"라고 잡히는데, 뭘 결정할 건지 모르겠을 때가 있어요.

**정우성**: 개발팀은 "No Meeting Wednesday"를 해봤는데 꽤 효과 있었어요. 수요일은 집중 개발 시간으로 지정하니까 생산성이 올라갔어요.

**김민수**: 전사적으로 확대해볼까요? 수요일 오전은 회의 금지로요.

**이서연**: 좋아요. 그리고 회의 요청 시 아젠다와 예상 결과물을 필수로 적게 하면 어떨까요?

### 결정 사항
- "No Meeting Wednesday Morning" 전사 도입 (수요일 9-12시 회의 금지)
- 회의 요청 시 필수 기재: 아젠다, 예상 결과물, 필수 참석자
- 30분 이내로 끝낼 수 있는 건 Slack 스레드로 대체

### 액션 아이템
- [ ] 오수현: 회의 가이드라인 문서 작성 및 전사 공지 (9/10까지)
- [ ] 김민수: 수요일 오전 회의 금지 캘린더 블락 (9/6까지)
