## 1. 스프린트 운영

### 스프린트 기본 정보
- **기간**: 2주
- **팀 벨로시티**: 약 40 스토리 포인트

> "총 47pt인데, 팀 벨로시티가 40pt 정도였잖아요. 조금 빡빡한데요?"
> — 박지현 CTO (2024-04 개발스프린트 회의)

### 스프린트 회의 일정

| 회의 | 시간 | 참석자 | 목적 |
|------|------|--------|------|
| 스프린트 플래닝 | 스프린트 첫날 09:30 | 개발팀 전체 | 백로그 리뷰, 태스크 배정 |
| 데일리 스탠드업 | 매일 09:30 | 개발팀 전체 | 진행 상황 공유 |
| 스프린트 회고 | 스프린트 마지막날 | 개발팀 전체 | 회고 및 개선점 도출 |

### 스프린트 플래닝 프로세스

1. **이전 스프린트 회고**
   - 완료율 확인
   - 이월 스토리 파악

2. **백로그 리뷰**
   - 스토리 포인트 산정
   - 우선순위 조정

3. **태스크 배정**
   - 담당자 지정
   - 예상 완료일 설정

> "Sprint 7 마무리부터 정리할게요. 계획했던 13개 스토리 중 11개 완료, 2개가 다음 스프린트로 이월됐습니다. 완료율 85%예요."
> — 정우성 개발팀장 (2024-04 개발스프린트 회의)

### 스토리 포인트 가이드

| 포인트 | 복잡도 | 예시 |
|--------|--------|------|
| 1-2pt | 간단한 수정 | 오타 수정, 색상 변경 |
| 3-5pt | 중간 복잡도 | 새 API 엔드포인트, UI 컴포넌트 |
| 8pt | 높은 복잡도 | 상태 관리 로직, 복잡한 쿼리 |
| 13pt | 매우 높은 복잡도 | 새로운 기능 모듈, 아키텍처 변경 |

## 2. 코드 리뷰 프로세스

### 기본 원칙
- **모든 코드는 PR(Pull Request)를 통해 머지**
- **최소 1명의 리뷰어 승인 필요**
- **리뷰는 24시간 내 완료 목표**

### PR 작성 가이드

```markdown
## 변경 사항
- [변경 내용 요약]

## 테스트
- [ ] 단위 테스트 통과
- [ ] 스테이징 테스트 완료

## 스크린샷 (UI 변경 시)
[스크린샷 첨부]
```

### 리뷰어 지정
- **백엔드 코드**: 백엔드 개발자 1명
- **프론트엔드 코드**: 프론트엔드 개발자 1명
- **크로스 기능**: 양쪽 각 1명

### 리뷰 시 확인사항

1. **코드 품질**
   - 가독성
   - 네이밍 컨벤션
   - 중복 코드 여부

2. **기능 동작**
   - 요구사항 충족
   - 엣지 케이스 처리

3. **성능**
   - N+1 쿼리 확인
   - 불필요한 렌더링

> "리뷰 완료! 몇 가지 코멘트 남겼어요. 특히 useEffect 의존성 배열 확인해주세요."
> — 이지영 (#dev 슬랙, 2024-01-15)

### 흔한 리뷰 피드백

| 항목 | 예시 |
|------|------|
| 의존성 배열 | "useEffect 의존성 배열 확인해주세요" |
| 쿼리 최적화 | "다음부터 쿼리 로그 확인하는 습관 들이면 돼요" |
| 가상화 적용 | "200개 넘는 케이스도 고려해주세요" |

## 3. 배포 프로세스

### 배포 환경

| 환경 | 용도 | URL |
|------|------|-----|
| Local | 개발 | localhost:3000 |
| Staging | 테스트 | staging.taskflow.io |
| Production | 운영 | app.taskflow.io |

### 정기 배포

- **시간**: 새벽 2-4시 (사용자 영향 최소화)
- **공지**: 최소 1일 전 #dev, #general 채널 공지

> "🔔 내일(4/26) 새벽 2-4시 사이 정기 점검 예정입니다. 해당 시간에 서비스 접속 불가해요~"
> — 김철수 개발팀장 (#general 슬랙, 2024-04-25)

### 배포 체크리스트

```markdown
## 배포 전
- [ ] PR 머지 완료
- [ ] 스테이징 테스트 통과
- [ ] 팀 공지 완료

## 배포 중
- [ ] 배포 스크립트 실행
- [ ] 헬스체크 확인

## 배포 후
- [ ] 모니터링 대시보드 확인 (Grafana)
- [ ] 에러율 확인
- [ ] 주요 기능 스모크 테스트
```

### 핫픽스 배포

긴급 버그 발생 시 핫픽스 프로세스:

1. **이슈 파악**: 원인 분석
2. **핫픽스 브랜치 생성**: `hotfix/issue-description`
3. **빠른 리뷰**: 1명 승인
4. **즉시 배포**
5. **모니터링**

> "핫픽스 완료. CPU 정상화됐어요. 죄송합니다 제가 리뷰 때 못 잡았네요."
> — 김철수 (#dev 슬랙, 2024-03-04)

### 배포 후 모니터링

- **에러율**: 0.1% 미만 유지
- **응답 시간**: 200ms 이하
- **CPU/메모리**: 80% 이하

> "3.0 런칭 후 모니터링 리포트 - 에러율: 0.01%, 평균 응답 시간: 150ms, AI 추천 사용률: 벌써 12%!"
> — 박민호 (#dev 슬랙, 2024-10-28)

## 4. 장애 대응 프로세스

### 장애 등급

| 등급 | 정의 | 대응 시간 |
|------|------|-----------|
| P1 (Critical) | 서비스 전체 불가 | 즉시 |
| P2 (High) | 핵심 기능 불가 | 1시간 내 |
| P3 (Medium) | 일부 기능 불가 | 4시간 내 |
| P4 (Low) | 사소한 이슈 | 다음 스프린트 |

### 장애 대응 절차

```
1. [감지] PagerDuty 알림 또는 고객 신고
    ↓
2. [공유] #dev 채널에 상황 공유
    ↓
3. [분석] 원인 파악 (로그, 쿼리, 모니터링)
    ↓
4. [해결] 핫픽스 배포 또는 롤백
    ↓
5. [안내] CS팀에 상황 전달 → 고객 안내
    ↓
6. [회고] 포스트모템 작성
```

### 실제 장애 대응 사례

**사례: 2024-03-04 새벽 CPU 급증**

```
[06:30] @봇 (PagerDuty): 🚨 ALERT: Production server high CPU usage (95%)
[06:35] @김철수: 확인 중이에요. 새벽부터 트래픽 급증...
[07:00] @김철수: 원인 파악! 어제 배포한 코드에서 N+1 쿼리 이슈 발생. 핫픽스 배포합니다.
[07:30] @김철수: ✅ 핫픽스 완료. CPU 정상화됐어요.
```

> "괜찮아요! 이런 거 다 배우는 거죠. 다음부터 쿼리 로그 확인하는 습관 들이면 돼요."
> — 김철수 (#dev 슬랙, 장애 후 피드백)

### 포스트모템 템플릿

```markdown
## 장애 요약
- 발생 시간:
- 해결 시간:
- 영향 범위:

## 원인
- 근본 원인:
- 직접 원인:

## 타임라인
- HH:MM - 장애 감지
- HH:MM - 원인 파악
- HH:MM - 해결

## 재발 방지 대책
- [ ] 조치 사항 1
- [ ] 조치 사항 2
```

## 7. 기술 부채 관리

### 기술 부채 유형

| 유형 | 예시 |
|------|------|
| 코드 품질 | 리팩토링 필요한 레거시 코드 |
| 아키텍처 | 상태 관리 불일치 (Redux + Context) |
| 문서화 | 온보딩 문서 부족 |
| 테스트 | 테스트 커버리지 부족 |

> "기존 REST API 코드가 좀 지저분해요. 리팩토링이 필요할 것 같습니다."
> — 김태호 (2024-04 개발스프린트 회의)

> "프론트엔드도 마찬가지예요. 상태 관리가 Redux랑 Context API가 섞여 있어서 헷갈려요."
> — 이하린 (2024-04 개발스프린트 회의)

### 기술 부채 스프린트

- **시점**: 메이저 버전 출시 후 (예: 2024년 8월)
- **기간**: 2주
- **내용**: 리팩토링 + 문서화

> "2.0 출시하고 나서 기술 부채 스프린트를 잡읍시다. 7월 출시 후 8월에 2주 정도 리팩토링 기간을 가지는 게 어떨까요?"
> — 박지현 CTO (2024-04 개발스프린트 회의)

### 기술 부채 리스트업

```markdown
## 백엔드
- [ ] API 리팩토링
- [ ] 쿼리 최적화

## 프론트엔드
- [ ] 상태 관리 통일 (Redux Toolkit)
- [ ] 컴포넌트 리팩토링

## 인프라
- [ ] 로그 로테이션 설정
- [ ] 모니터링 강화
```

### 스레드 활용

긴 논의는 스레드 사용:

```
메인 메시지: "PR 올렸습니다"
└ 스레드: 리뷰 코멘트
└ 스레드: 수정 사항
└ 스레드: 머지 완료
```

### 긴급 이슈 공유 형식

```markdown
🚨 [긴급] 이슈 제목

**현상**: [설명]
**영향**: [범위]
**현재 상태**: [대응 중/해결/모니터링]
```

## 10. 보안 가이드라인

### 보안 감사 결과 반영 사항

| 항목 | 조치 |
|------|------|
| 세션 타임아웃 | 7일 → 24시간 (Remember me로 7일 유지 가능) |
| API 레이트 리미팅 | 분당 100회 제한 |
| 로그 마스킹 | 이메일 등 PII 데이터 마스킹 |

> "세션 타임아웃이 너무 길어요. 현재 7일인데, 권고사항은 최대 24시간입니다."
> — 박지현 CTO (2024-08 보안점검 회의)

### 코드 보안 체크리스트

```markdown
## PR 시 확인
- [ ] 민감 정보 하드코딩 없음
- [ ] SQL 인젝션 방지
- [ ] XSS 방지
- [ ] 적절한 권한 체크
```

### 보안 교육

- **분기 1회**: 내부 보안 교육 세션
- **연 1회**: 외부 보안 교육 (OWASP 등)

> "분기 1회 내부 보안 교육 세션 도입 (9월 시작)"
> — 2024-08 보안점검 회의
